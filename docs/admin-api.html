Admin API

Five HTTP endpoints that give you runtime insight and live control over the rate limiter — without touching the collector process or restarting anything.

Enabling the Admin Server

Add admin_addr to your processor config to expose the API:

processors:
  ratelimiter:
    policy_file: /etc/otelcol/policies.json
    admin_addr: ":8090"   # listens on all interfaces, port 8090
Security: Bind to loopback (127.0.0.1:8090) in production, or place the admin port behind an authenticated reverse proxy. The API has no built-in authentication.

Endpoints

MethodPathPurpose
GET /health Service liveness and uptime. Returns 200 OK when the processor is running.
GET /stats Cumulative per-signal counters: received, allowed, dropped, denied, shadow, errors, reloads.
GET /config Sanitized view of the current running configuration. Useful for verifying a hot reload applied correctly.
POST /reload Force an immediate policy reload across all engines. Returns the number of engines refreshed.
GET /metrics Prometheus text format metrics — the same counters as /stats but in TYPE/HELP Prometheus format.

Usage Examples

Check processor health

curl -s http://localhost:8090/health

Inspect live counters

curl -s http://localhost:8090/stats | jq .

See active policy config

curl -s http://localhost:8090/config | jq .

Force a policy reload

curl -s -X POST http://localhost:8090/reload | jq .

Scrape Prometheus metrics

curl -s http://localhost:8090/metrics

Operational Quick Checks

During an incident

Use /stats to see if drop rates are abnormally high, and /config to confirm the policy that's active — no Prometheus lag.

After a policy change

Call POST /reload then immediately check /config to verify the new policy loaded successfully before checking metrics.

Kubernetes readiness probe

Point your readinessProbe.httpGet at /health on port 8090 — returns non-200 if the processor is not ready to serve traffic.

No Prometheus? No problem

GET /stats gives cumulative totals instantly. Useful for shell scripts, CI checks, and on-call runbooks.

Metrics from /metrics are documented in detail on the Monitoring page, including PromQL examples and alert rules.
