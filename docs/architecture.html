Full System Diagram

Telemetry enters from any OTLP-compatible source, passes through the Rate Limiter Processor where every record is evaluated against policies, and exits to your chosen backend — with full metrics and live control throughout.

All components run inside a single OTEL Collector binary — no extra network hops or sidecars required.

Inside the Processor — What Each Component Does

Context Builder

Extracts signal type (logs / traces / metrics), service name, and relevant attributes from each incoming record to build a rich evaluation key for the policy engine.

Policy Evaluator (RLAAS)

Runs the configured rate-limiting algorithm (e.g. token_bucket) against the context. Returns an allow, deny, or shadow verdict per record.

Decision Engine

Applies the verdict: allowed records pass through, denied records are dropped and counted, shadow records are counted as "would deny" but still forwarded — safe for dry-run testing.

Batch Guard

A hard pre-check: if the batch size exceeds max_batch_size, excess records are dropped before evaluation even begins — protecting CPU and memory on extreme bursts.

OTEL Metrics

Emits seven Prometheus-compatible counters per signal type: received, allowed, dropped, denied, shadow, evaluate_errors, and policy_reloads.

Admin API

Five HTTP endpoints — /health, /stats, /config, /reload, /metrics — give operators runtime insight and live control without touching the collector process.

Behavior Guarantees

Fail-open by default Set fail_open: true so that if the policy engine errors, records continue flowing — no silent data loss during evaluation failures.
Shadow mode for safe rollouts Mark any policy as action: shadow_only to count what would be dropped without actually dropping it. Verify impact in Grafana before enforcing.
Fail-closed for compliance paths Set fail_open: false on high-sensitivity pipelines to guarantee no unevaluated data reaches exporters during an engine error.
