Configuration Reference

All fields supported by the current processor implementation.

processors:
  ratelimiter:
    policy_file: /etc/otelcol/policies.json
    # policies_inline: |
    #   [...]
    fail_open: true
    cache_ttl: 30s
    watch_policies: false
    watch_interval: 15s
    key_prefix: otel
    org_id: ""
    tenant_id: ""
    application: ""
    environment: ""
    service_expr: ""
    org_id_expr: ""
    tenant_id_expr: ""
    application_expr: ""
    environment_expr: ""
    max_batch_size: 0
    admin_addr: ""
    admin_auth_token: ""
    admin_token_header: "Authorization"

Validation Rules

  • Exactly one of policy_file or policies_inline must be set.
  • cache_ttl, watch_interval, and max_batch_size cannot be negative.
  • If TLS is enabled, both admin_tls_cert_file and admin_tls_key_file are required.

Secrets and Vault Integration

No secrets ever need to be hardcoded in collector configuration. The OTEL Collector's built-in environment variable expansion lets you inject tokens, paths, and credentials from any secret store.

Environment variable expansion

The OTEL Collector resolves ${env:VAR_NAME} at startup for all string config fields:

processors:
  ratelimiter:
    policy_file: /etc/otelcol/policies.json
    admin_addr: ":8090"
    admin_auth_token: "${env:OTEL_ADMIN_TOKEN}"
    admin_tls_cert_file: "${env:ADMIN_TLS_CERT}"
    admin_tls_key_file: "${env:ADMIN_TLS_KEY}"
    org_id: "${env:ORG_ID}"
Tip: Set environment variables in your process supervisor, Kubernetes Secret, or Vault Agent before launching the collector. The collector never reloads env vars at runtime — restart is required to pick up changes to environment variables.

Policy file from any path

policy_file is a plain file system path. It works with any mechanism that delivers files:

Vault Agent sidecar

Vault Agent renders the policy file from a Vault secret to a shared volume. The collector reads from the volume path; Vault Agent keeps it fresh.

# Vault Agent template
{{ with secret "secret/otel/policies" }}
{{ .Data.data.json | toJSON }}
{{ end }}

# Collector config
policy_file: /vault/secrets/policies.json
watch_policies: true

Kubernetes ConfigMap

Mount a ConfigMap as a file. Use watch_policies: true to hot-reload when the ConfigMap is updated (kubelet typically propagates within 60s).

# ConfigMap mount in pod spec
volumes:
  - name: policies
    configMap:
      name: ratelimiter-policies
volumeMounts:
  - name: policies
    mountPath: /etc/otelcol

# Collector config
policy_file: /etc/otelcol/policies.json
watch_policies: true

AWS Secrets Manager / SSM

Use the AWS Secrets Manager Agent or a sidecar (e.g., aws-secrets-manager-agent) to write secrets to a file. Point policy_file at that file.

# Example with secrets-store-csi-driver
volumes:
  - name: policies-secret
    csi:
      driver: secrets-store.csi.k8s.io
      readOnly: true
      volumeAttributes:
        secretProviderClass: otel-policies

Inline from CI/CD pipeline

Inject policy JSON directly via policies_inline using an environment variable expanded at deploy time — no file on disk needed.

# In your deployment pipeline:
export OTEL_POLICIES_INLINE=$(cat policies.json)

# Collector config:
processors:
  ratelimiter:
    policies_inline: "${env:OTEL_POLICIES_INLINE}"

Hot reload after secret rotation

When a sidecar updates the policy file, the collector picks up the change automatically if watch_policies: true. For controlled rollouts, disable file watching and call POST /reload explicitly after your sidecar confirms the file is written:

# In a post-rotation hook:
curl -s -X POST https://collector:8090/reload \
  -H "Authorization: Bearer ${OTEL_ADMIN_TOKEN}"
# Confirm the reload applied the new SHA
curl -s https://collector:8090/config/policies | jq '.content_sha256'

Admin token security

The admin auth token is a shared secret. Rotate it by:

  1. Updating the secret in Vault / Kubernetes Secret.
  2. Rolling the collector pod (or restarting the process) to pick up the new ${env:OTEL_ADMIN_TOKEN} value.
  3. Updating any automation or dashboards that call the Admin API with the new token.
