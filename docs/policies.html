Policy Reference

How policies are defined, scoped, matched, and applied — from a single global default to per-service per-org precision limits.

Policy Schema

A policy file is a JSON array. Each object in the array defines one rate-limit rule.

[
  {
    "id":                "checkout-logs-prod",
    "name":              "Checkout logs — production limit",
    "enabled":           true,
    "signal_type":       "log",
    "service":           "checkout",
    "org_id":            "acme-corp",
    "tenant_id":         "tenant-42",
    "application":       "storefront",
    "environment":       "production",
    "algorithm":         "token_bucket",
    "rate":              500,
    "burst":             1000,
    "action":            "drop",
    "enforcement_mode":  "enforce"
  }
]
FieldTypeRequiredDescription
idstringyesUnique identifier for this policy; appears in /stats matched policy field.
namestringnoHuman-readable label. Shown in Admin UI and logs.
enabledboolnoDefault true. Set false to disable without deleting.
signal_typestringnolog, span, or metric. Omit to match all signals.
servicestringnoService name to match. Omit for org-wide or global scope.
org_idstringnoOrganization identifier. Omit for global scope.
tenant_idstringnoTenant identifier for multi-tenant deployments.
applicationstringnoApplication name scope. Omit to match any application.
environmentstringnoproduction, staging, etc. Omit to match all environments.
algorithmstringyesRate limit algorithm. See algorithms section below.
rateintyesAllowed records per window/second.
burstintnoBurst headroom above rate. Applicable to token_bucket.
actionstringyesdrop — remove over-limit records; allow — pass all (for exceptions).
enforcement_modestringnoenforce (default) or shadow_only — observe without dropping.

Rate Limit Algorithms

token_bucket

Tokens refill at rate per second. Burst allows short spikes above the steady-state rate. Best for general-purpose throttling.

"algorithm": "token_bucket",
"rate": 500,
"burst": 1000

fixed_window

Counts records in a fixed time window (e.g., 1 minute). Resets at window boundary. Simple and predictable.

"algorithm": "fixed_window",
"rate": 10000

sliding_window

Smoothed count over the past N seconds. Prevents burst spikes at window edges; more accurate than fixed window.

"algorithm": "sliding_window",
"rate": 10000

leaky_bucket

Records drain at a constant rate; excess queues or drops. Good for enforcing strict steady-state throughput.

"algorithm": "leaky_bucket",
"rate": 200

Service and Org Scoping

Every telemetry record carries context extracted from its OTEL attributes. The policy engine matches the most specific policy that fits that context.

The processor extracts these context fields from every incoming record before evaluating it against policies:

Context FieldExtracted FromConfig Override
service Auto: resource.attributes["service.name"]
Custom: service_expr		 Expression overrides the auto lookup. See Expression Language.
org_id org_id_expr → fallback to org_id config value Set org_id: "acme" to inject a default for all records.
tenant_id tenant_id_expr → fallback to tenant_id config value Useful for SaaS platforms with per-tenant billing.
application application_expr → fallback to application config value Group multiple services under one application umbrella.
environment environment_expr → fallback to environment config value Set environment: "production" to tag all records.
signal_type Determined by pipeline: log, span, metric Cannot be overridden — set by which pipeline processes the record.

Fallback Hierarchy

When evaluating a record, the RLAAS engine finds the most specific matching policy. If none match at that specificity level, it falls through to broader policies.

1
Service-specific + Org match
Policy has both service: "checkout" and org_id: "acme". Most precise — applies to one service within one org. 
{ "service": "checkout", "org_id": "acme", "rate": 500 }
2
Service-specific (any org)
Policy has service: "checkout" but no org_id. Applies to that service across all orgs. 
{ "service": "checkout", "rate": 500 }
3
Org-level default
Policy has org_id: "acme" but no service. Applies to all services in that org that have no more specific policy. 
{ "org_id": "acme", "rate": 2000 }
4
Signal-type default
Policy has signal_type: "log" but no service or org. Applies to all log records that haven't matched anything above. 
{ "signal_type": "log", "rate": 10000 }
5
Global default
Policy has no scope fields at all. Catches everything. Acts as the catch-all safety net. 
{ "id": "global-default", "rate": 50000, "action": "drop" }
The engine always picks the most specific matching policy. If a record matches both a service policy and an org policy, the service policy wins. Only one policy is applied per record.

Complete layered example

[
  // Global safety net — catches everything with no other policy
  {
    "id":          "global-default",
    "algorithm":   "token_bucket",
    "rate":        50000,
    "burst":       75000,
    "action":      "drop"
  },
  // Org-level default — all ACME services that have no specific rule
  {
    "id":          "acme-default",
    "org_id":      "acme",
    "algorithm":   "token_bucket",
    "rate":        5000,
    "burst":       8000,
    "action":      "drop"
  },
  // Service-specific for ACME — most precise, wins over acme-default
  {
    "id":          "acme-checkout-logs",
    "org_id":      "acme",
    "service":     "checkout",
    "signal_type": "log",
    "algorithm":   "token_bucket",
    "rate":        500,
    "burst":       1000,
    "action":      "drop"
  }
]

Expression Language

The expression language lets you extract context fields from arbitrary OTEL attributes without changing the collector code. Expressions are configured in the processor YAML.

Expressions use || to define a priority-ordered list of attribute lookups. The first non-empty value wins:

service_expr:     "resource.attributes.service.name || attributes.service || k8s.deployment.name"
org_id_expr:      "resource.attributes.org.id || attributes.org_id || attributes.x-org-id"
tenant_id_expr:   "attributes.tenant.id || attributes.tenant_id || attributes.x-tenant-id"
application_expr: "attributes.app.name || resource.attributes.service.name"
environment_expr: "attributes.deployment.environment || resource.attributes.deployment.environment || \"production\""
Token FormLooks UpExample
resource.attributes.KEY Resource-level OTEL attribute (set by the SDK or Collector) resource.attributes.service.name
attributes.KEY Span / log / metric record-level attribute attributes.tenant_id
tags.KEY Alias for attributes.KEY tags.org.id
KEY (bare) Searches merged attributes then resource attributes k8s.deployment.name
"literal" or 'literal' Static string — used as a final fallback default "production"
Expressions are evaluated left to right. Put the most specific / most common attribute first. Put quoted literals last — they always match, so anything after them is unreachable.

How attributes flow through the pipeline

The processor merges resource-level and record-level attributes into a single flat map before evaluating expressions. Record attributes override resource attributes on key collisions.

Hot Reload

Policy files can be updated without restarting the collector.

File-system watching

Enable watch_policies: true to detect file changes automatically using fsnotify. Changes apply within milliseconds on local file systems.

watch_policies: true
watch_interval: 15s  # polling fallback

watch_interval is a polling fallback for NFS, Docker bind-mounts, and network file systems where fsnotify events may be unreliable.

Manual reload via API

Call POST /reload on the Admin API to force an immediate reload across all engines. Useful for deterministic rollout windows — publish the file, then trigger reload.

curl -s -X POST http://localhost:8090/reload \
  -H "Authorization: Bearer $TOKEN"

Inline policies (policies_inline) require a full collector restart — hot reload only applies to policy_file.

Safe Rollout Strategy

Never enforce a new limit cold. Use shadow mode to observe impact before dropping traffic.

1
Add policy in shadow mode — records count toward the limit but are never dropped. Watch ratelimiter_records_shadow_total in Prometheus to see how many would have been denied. 
{ "id": "payments-shadow", "enforcement_mode": "shadow_only",
  "service": "payments", "rate": 80, "burst": 150 }
2
Observe for 24–48 hours — confirm the shadow counter stays below your acceptable drop rate. Tune rate/burst if the shadow rate is too high.
3
Switch to enforce — change enforcement_mode from shadow_only to enforce. The counter moves from shadow to denied. 
{ "id": "payments-shadow", "enforcement_mode": "enforce",
  "service": "payments", "rate": 80, "burst": 150 }
4
Trigger reload — either wait for watch_policies to pick up the change, or call POST /reload immediately.
5
Monitor for 15 minutes — confirm ratelimiter_records_denied_total is rising at the expected rate and downstream services are healthy.

Common Policy Patterns

Per-service log limit with org scope

[
  {
    "id":          "checkout-acme-logs",
    "org_id":      "acme",
    "service":     "checkout",
    "signal_type": "log",
    "algorithm":   "token_bucket",
    "rate":        500,
    "burst":       1000,
    "action":      "drop"
  }
]

Global trace limit (all services, all orgs)

[
  {
    "id":          "global-trace-limit",
    "signal_type": "span",
    "algorithm":   "token_bucket",
    "rate":        20000,
    "burst":       40000,
    "action":      "drop"
  }
]

Multi-tenant with per-org defaults

[
  { "id": "org-free-tier",  "org_id": "free",  "rate": 1000,  "burst": 2000,  "action": "drop" },
  { "id": "org-pro-tier",   "org_id": "pro",   "rate": 10000, "burst": 20000, "action": "drop" },
  { "id": "org-enterprise", "org_id": "ent",   "rate": 100000,"burst": 150000,"action": "drop" },
  { "id": "global-default", "rate": 500, "burst": 1000, "action": "drop" }
]

Exception policy (allow-list)

[
  {
    "id":      "internal-monitoring-exempt",
    "service": "otel-collector-self",
    "action":  "allow",
    "rate":    999999
  }
]
Use GET /config/policies to confirm the exact policy payload the engine is running. The response includes SHA-256 checksum and file modification timestamp so you can verify a reload applied correctly.
