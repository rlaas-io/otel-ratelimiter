Policy Design

The processor relies on RLAAS policy evaluation. Keep policies narrow, observable, and staged with shadow mode.

Baseline policy

[
  {
    "id": "checkout-logs",
    "name": "Checkout logs limit",
    "signal_type": "log",
    "service": "checkout",
    "algorithm": "token_bucket",
    "rate": 500,
    "burst": 1000,
    "action": "drop"
  }
]

Safe rollout policy

[
  {
    "id": "payments-shadow",
    "name": "Observe before enforce",
    "signal_type": "trace",
    "service": "payments",
    "algorithm": "token_bucket",
    "rate": 80,
    "burst": 150,
    "action": "shadow_only"
  }
]

Practical guidance

  • Start with expected steady-state throughput for rate.
  • Set burst between 1.5x and 3x of rate for short spikes.
  • Prefer service-specific policies over global wildcards for critical paths.
  • Use shadow mode first, then switch to enforced drop once validated.
With watch_policies: true, policy updates are applied live. For deterministic rollout windows, use POST /reload after file publish.
