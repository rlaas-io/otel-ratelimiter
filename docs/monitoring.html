Monitoring & Alerting

Seven counters cover every decision path. Use both the OTEL metrics endpoint and the Admin API for complete operational visibility — no blind spots.

What You Can See

Every record processed by the rate limiter increments exactly one counter per path. Together they give a complete picture of what entered, what was approved, and what was rejected — broken down by signal type.

ratelimiter_records_received_total

Total records that entered the processor. The baseline for computing all ratios.

ratelimiter_records_allowed_total

Records approved by policy and forwarded to the exporter.

ratelimiter_records_dropped_total

Records dropped for any reason — rate limit exceeded, batch cap, or evaluation error in fail-closed mode.

ratelimiter_records_denied_total

Records explicitly denied by a policy rule (excludes batch-cap drops).

ratelimiter_records_shadow_total

Records that would have been denied but were forwarded because the policy is in shadow_only mode. Use to validate new policies safely.

ratelimiter_evaluate_errors_total

Policy evaluation errors. Non-zero means the engine had trouble evaluating — investigate immediately.

ratelimiter_policy_reloads_total

Number of successful policy reloads. Unexpected spikes may indicate a config management issue.

All counters carry a signal label (logs, traces, metrics) so you can monitor each telemetry type independently.

Useful PromQL Queries

Drop rate by signal (per minute)

sum by (signal) (rate(ratelimiter_records_dropped_total[5m]))

Allow rate by signal (per minute)

sum by (signal) (rate(ratelimiter_records_allowed_total[5m]))

Drop ratio — percentage of traffic being dropped

sum by (signal) (rate(ratelimiter_records_dropped_total[5m]))
/
sum by (signal) (rate(ratelimiter_records_received_total[5m]))

Evaluation error rate (should be zero)

sum by (signal) (rate(ratelimiter_evaluate_errors_total[5m]))

Shadow mode — what would be denied (policy dry-run)

sum by (signal) (rate(ratelimiter_records_shadow_total[5m]))

Policy reload count in the last hour

sum by (signal) (increase(ratelimiter_policy_reloads_total[1h]))

Alert Rules

Start with these three alerts. Tune thresholds for your traffic profile.

  • 🔴
    Evaluation errors > 0 for 5 minutes
    The policy engine is failing to evaluate records. Check policy file validity and RLAAS connectivity. Fire immediately.
  • 🟡
    Drop ratio > 20% for 10 minutes
    More than one in five records is being dropped. May be intentional (tight budget) or a sign that a noisy service needs its own policy.
  • 🟡
    Policy reloads > 10 in 10 minutes
    Unexpected reload spikes usually indicate a config management loop — a CI system or operator script repeatedly writing the policy file.

Quick Health Check via Admin API

No Prometheus? Use the Admin API for an instant sanity check:

# Is the processor alive?
curl -s http://otel-collector:8090/health

# What are the live totals?
curl -s http://otel-collector:8090/stats | jq .

# What policy is currently active?
curl -s http://otel-collector:8090/config | jq .
Tip: GET /stats returns cumulative counters since startup — useful for spotting absolute totals during an incident when Prometheus lag matters.
